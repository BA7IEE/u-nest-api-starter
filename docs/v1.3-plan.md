# V1.3 Contract Hardening Plan

> 状态:**仅文档,不执行开发**。
> 立项前提:V1.2 / v0.1.3 已合并并打 tag。
> 范围承诺:V1.3 仍**不**新增业务功能、不动 `prisma/schema.prisma`、不改 14 个业务接口的路径 / 方法 / 入参 / 出参。
> 升级条件:见 [`ARCHITECTURE.md`](../ARCHITECTURE.md) §9。

---

## 0. 一句话目标

让"模板的契约面"(API schema、错误码 ↔ HTTP status、权限策略、依赖版本)从"E2E 顺带覆盖"升级为"独立断言 + 自动化护栏",降低未来业务模块进来时无意识破坏 v1 约束的概率。

---

## 1. OpenAPI 契约快照测试

### 现状

`/api/docs-json` 已暴露完整 OpenAPI v3 schema,但仓库无任何快照文件。任何 controller / DTO 的"无意识"修改(字段重命名、类型变化、`@ApiProperty` 描述漂移)只能靠 code review 肉眼发现。

### V1.3 提议

- 新增 `test/contract/openapi.snapshot.spec.ts`,启动 NestJS 应用后取 `/api/docs-json`,与 `test/contract/__snapshots__/openapi.json` 比对
- 快照文件**进 git**,任何 schema 变更必须显式 `pnpm test:contract -u`,在 PR diff 里能直接 review
- 新增 `pnpm test:contract` 脚本,与 `test:e2e` 并列(也可作为 `test:e2e` 的一部分,按运行时长决定)
- CI 中 `pnpm test:contract` 失败即阻塞合并

### 不做

- 不引入 `dredd` / `prism` / 其他 mock 服务器
- 不强制语义版本号绑定(`patch` 也可以是 schema 变更,只要快照同步更新)

## 2. `users.policy.ts` 单元测试矩阵

### 现状

V1.2 提取出 4 个纯函数(`canViewUser` / `canManageUser` / `canCreateRole` / `canChangeRole`),但仅靠 E2E 间接覆盖。3 种 actorRole × 3 种 targetRole = 9 个组合 × 4 函数 = **36 个判定点**,目前没有单测直接断言。

### V1.3 提议

- 新增 `src/modules/users/users.policy.spec.ts`(纯单元,不启动 Nest)
- 用 `it.each` 矩阵覆盖 36 个组合,断言每个函数的布尔输出
- `pnpm test:unit`(新脚本)与 `pnpm test:e2e` 分离;CI 都跑

### 收益

- 修改策略时秒级反馈,无需起 Postgres
- 36 个判定点显式锁定,未来若新增第 5 个函数(如 `canResetPassword`)有现成模板

## 3. Prisma 6 → 7 升级评估

### 现状

`@prisma/client@^6.19.3`;Prisma CLI 启动时打印 `Update available 6.19.3 -> 7.8.0`。Prisma 7 的[已知 breaking changes](https://www.prisma.io/docs/orm/more/upgrade-guides/upgrading-versions) 包括:

- `package.json#prisma` 配置块在 7.x 移除,必须迁到 `prisma.config.ts`(本仓库已收到 deprecation warning)
- 默认 `engineType` 调整(可能影响 Docker 镜像层 / 启动时间)
- `Prisma.PrismaClientKnownRequestError` 字段语义有微调,需要确认 §5 P2002 处理仍正确

### V1.3 提议

- 单独立项 `phase-15.6-prisma-7-upgrade`(参考 `TASKS.md` 编号习惯)
- 步骤:
  1. `pnpm add prisma@7 @prisma/client@7`
  2. 创建 `prisma.config.ts`,迁移 `seed` 配置
  3. 跑 `pnpm prisma:generate` + `pnpm test:e2e`
  4. 重新构建 Docker 镜像,确认 runner 阶段产物大小无回退
  5. 更新 `docs/development.md` 与 `docs/deployment.md` 中 Prisma 版本号

### 不做

- 不在 V1.3 主线里隐式升级;严格走独立任务

## 4. BizCode ↔ HTTP status 联表自动断言

### 现状

[`test/e2e/bizcode-http-status.e2e-spec.ts`](../test/e2e/bizcode-http-status.e2e-spec.ts) 已覆盖部分 BizCode 的 HTTP status 一致性,但**遍历不完整**:新增 BizCode 时若忘记加测试用例,直接漏网。

### V1.3 提议

- 新增 `src/common/exceptions/biz-code.spec.ts`(纯单元),`Object.values(BizCode)` 遍历断言每个条目都满足:
  - `code` 数字 ∈ 已分段范围(`4xxxx`/`5xxxx`/`100xx`/`101xx`/`110xx+`)
  - `httpStatus` ∈ NestJS `HttpStatus` 枚举
  - `code` / `message` / `httpStatus` 三字段全部存在且非空
- E2E 层保留典型路径断言(`UNAUTHORIZED` / `FORBIDDEN_ROLE_OPERATION` / `LAST_SUPER_ADMIN_PROTECTED` 等),不重复机械遍历

## 5. 错误响应 schema 显式化

### 现状

[`api-response.decorator.ts`](../src/common/decorators/api-response.decorator.ts) 提供 `@ApiWrappedOkResponse` / `@ApiWrappedArrayResponse` / `@ApiWrappedPageResponse` 三个 success path 装饰器。错误响应在 OpenAPI schema 里仅靠 NestJS 内置默认 4xx/5xx 描述,前端 SDK 生成器拿不到 `{ code, message, data: null }` 结构。

### V1.3 提议

- 新增 `@ApiBizErrorResponse(httpStatus, ...bizCodes)` 装饰器
- 给所有 controller 方法补全错误响应注解(参考 BizCode 段位 4xxxx/5xxxx 通用 + 各模块业务错误 100xx/101xx)
- OpenAPI snapshot 测试(§1)会顺带锁定错误响应 schema

### 不做

- 不改 `AllExceptionsFilter` 行为
- 不改 BizCode 段位划分

## 6. Lint warnings 收敛到 0

### 现状

`pnpm lint` 输出 128 个 warnings,全部位于 `test/` 工具链(`@typescript-eslint/no-unsafe-argument`,主要在 `call-endpoint.ts` / fixtures / 各 spec 调用 `request(app.getHttpServer())` 时)。

### V1.3 提议

- 在 [`test/helpers/call-endpoint.ts`](../test/helpers/call-endpoint.ts) / fixtures 里收敛 `app: any` 到 `INestApplication`
- 或在 `eslint.config.mjs` 中针对 `test/**/*.ts` 关闭 `no-unsafe-argument`(明确选择,而非默认继承)
- 目标:`pnpm lint` 输出 `0 problems`,CI 改为 `--max-warnings 0` 严格模式

## 7. 不在 V1.3 范围

继续沿用 [`ARCHITECTURE.md`](../ARCHITECTURE.md) §9 升级条件;以下能力**仍然不做**:

- RBAC / 权限表 / 菜单权限 / `casl`
- refresh token / 本人改密码 / 强制下线
- Redis / BullMQ / 队列 / 定时任务
- 文件上传 Provider 实现(本地 / OSS / R2)
- 微信小程序登录 / OAuth 第三方登录 / 多租户 / 组织树
- LLM / 向量检索(`modules/ai/` 仍 README 占位)
- 操作日志 / 登录日志 / 审计日志的数据库持久化
- OpenTelemetry / Sentry / Datadog / APM / Prometheus `/metrics`
- `docker-compose.prod.yml` / K8s manifests / 镜像推送脚本

发现需求超出 V1.3 范围 → **必须暂停说明**,不得"先实现再回滚"。
