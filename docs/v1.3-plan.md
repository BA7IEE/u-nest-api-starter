# V1.3 Contract Hardening Plan

> 状态:**仅文档,不执行开发**。

> 范围承诺:V1.3 仍**不**新增业务功能、不动 `prisma/schema.prisma`、不改 14 个业务接口的路径 / 方法 / 入参 / 出参。

## 0. 一句话目标

让"模板的契约面"(API schema、错误码 ↔ HTTP status、权限策略、依赖版本)从"E2E 顺带覆盖"升级为"独立断言 + 自动化护栏",降低未来业务模块进来时无意识破坏 v1 约束的概率。

## 1. OpenAPI 契约快照测试 [done — V1.3-3]

### 现状

`/api/docs-json` 已暴露完整 OpenAPI v3 schema,但仓库无任何快照文件。任何 controller / DTO 的"无意识"修改(字段重命名、类型变化、`@ApiProperty` 描述漂移)只能靠 code review 肉眼发现。

### V1.3 提议

- 新增 `test/contract/openapi.snapshot.spec.ts`,启动 NestJS 应用后取 `/api/docs-json`,与 `test/contract/__snapshots__/openapi.json` 比对
- 快照文件**进 git**,任何 schema 变更必须显式 `pnpm test:contract -u`,在 PR diff 里能直接 review
- 新增 `pnpm test:contract` 脚本,与 `test:e2e` 并列(也可作为 `test:e2e` 的一部分,按运行时长决定)
- CI 中 `pnpm test:contract` 失败即阻塞合并

### 实现落点(V1.3-3)

- 实际文件名为 `test/contract/openapi.contract-spec.ts`(后缀与 `jest-contract.config.ts#testRegex` 对齐),配套 `test/contract/__snapshots__/openapi.contract-spec.ts.snap` 入 git ✅
- 显式断言 14 条路由(`POST /api/auth/login` + 3 个 health + 10 个 users)与全量路由白名单一致(防漏增) ✅
- 显式断言 11 个核心 DTO 仍出现在 `components.schemas`(LoginDto / LoginResponseDto / CreateUserDto / UpdateUserDto / UpdateMyProfileDto / UpdateUserRoleDto / UpdateUserStatusDto / ResetUserPasswordDto / UserResponseDto / HealthResponseDto / PageResultDto) ✅
- `paths` 与 `components.schemas` 两段独立 `toMatchSnapshot()`,故意排除 `info.version`(随发布递增,不视作 schema 漂移) ✅
- 新增 `test/jest-contract.config.ts`,复用 e2e 的 `global-setup` / `setup-files`,串行执行 ✅
- 新增 `pnpm test:contract` 脚本,验收门槛与 lint / typecheck / test / test:e2e / build 并列 ✅

### 不做

- 不引入 `dredd` / `prism` / 其他 mock 服务器
- 不强制语义版本号绑定(`patch` 也可以是 schema 变更,只要快照同步更新)

## 3. Prisma 6 → 7 升级评估

### 现状

`@prisma/client@^6.19.3`;Prisma CLI 启动时打印 `Update available 6.19.3 -> 7.8.0`。Prisma 7 的[已知 breaking changes](https://www.prisma.io/docs/orm/more/upgrade-guides/upgrading-versions) 包括:

- `package.json#prisma` 配置块在 7.x 移除,必须迁到 `prisma.config.ts`(本仓库已收到 deprecation warning)
- 默认 `engineType` 调整(可能影响 Docker 镜像层 / 启动时间)
- `Prisma.PrismaClientKnownRequestError` 字段语义有微调,需要确认 §5 P2002 处理仍正确

### V1.3 提议

- 单独立项 `phase-15.6-prisma-7-upgrade`(参考 `TASKS.md` 编号习惯)
- 步骤:
  1. `pnpm add prisma@7 @prisma/client@7`
  2. 创建 `prisma.config.ts`,迁移 `seed` 配置
  3. 跑 `pnpm prisma:generate` + `pnpm test:e2e`
  4. 重新构建 Docker 镜像,确认 runner 阶段产物大小无回退
  5. 更新 `docs/development.md` 与 `docs/deployment.md` 中 Prisma 版本号

### 不做

- 不在 V1.3 主线里隐式升级;严格走独立任务

## 5. 错误响应 schema 显式化 [done — V1.3-4]

### 现状

[`api-response.decorator.ts`](../src/common/decorators/api-response.decorator.ts) 提供 `@ApiWrappedOkResponse` / `@ApiWrappedArrayResponse` / `@ApiWrappedPageResponse` 三个 success path 装饰器。错误响应在 OpenAPI schema 里仅靠 NestJS 内置默认 4xx/5xx 描述,前端 SDK 生成器拿不到 `{ code, message, data: null }` 结构。

### V1.3 提议

- 新增 `@ApiBizErrorResponse(httpStatus, ...bizCodes)` 装饰器
- 给所有 controller 方法补全错误响应注解(参考 BizCode 段位 4xxxx/5xxxx 通用 + 各模块业务错误 100xx/101xx)
- OpenAPI snapshot 测试(§1)会顺带锁定错误响应 schema

### 实现落点(V1.3-4)

- 新增 `ApiBizErrorResponse(...bizCodes: BizCodeEntry[])`,签名只接 `BizCodeEntry`(不接裸 number 或字符串),类型层面强制只能引用 `BizCode` 常量 ✅
- 内部按 `httpStatus` 自动分组:同一 status 下多个 BizCode 合并到一条 `@ApiResponse`,`code.enum` 列出全部可能业务码,`code.description` 用 `code: message` 格式描述每一个,与真实运行时 `{ code, message, data: null }` 完全一致 ✅
- 给 14 条业务路由补完错误响应注解;auth/login 删掉原裸 `@ApiResponse({ status: 429 })`,统一走新装饰器 ✅
- OpenAPI snapshot 自动覆盖新增的错误响应 schema(`pnpm test:contract` 通过) ✅
- 不改 `AllExceptionsFilter`、不改 BizCode 段位、不改任何运行时响应行为 ✅

### 不做

- 不改 `AllExceptionsFilter` 行为
- 不改 BizCode 段位划分

## 5.5 CI 护栏补全 [done — V1.3-5]

### 现状(V1.3-4 合并后发现)

`.github/workflows/ci.yml` 的 `Lint / Typecheck / E2E` job 只跑 `lint / typecheck / build / prisma:deploy / test:e2e`,**没有 `pnpm test`(单元)与 `pnpm test:contract`(契约快照)**。这意味着 V1.3-1(`users.policy.spec.ts`)、V1.3-2(`biz-code.constant.spec.ts`)、V1.3-3 + V1.3-4(OpenAPI 契约快照含错误响应 schema)在 CI 内**没有真实护栏**——本地通过,CI 永远不会因为这两类测试失败而拦截 PR。

### 实现落点(V1.3-5)

- 在 `Verify production migration command` 之后、`Run E2E tests` 之前插入两步:
  - `Run unit tests` → `pnpm test`
  - `Run contract tests` → `pnpm test:contract` ✅
- 不动 `docker-build` job,不引入 smoke test(沿用 V1.1.1 显式选择) ✅
- 不动 `pnpm test:e2e` 既有顺序与触发条件 ✅

### 不做

- 不引入 Docker container smoke test
- 不改 lint `--max-warnings` 严格模式(留给 §6)
- 不改 ci.yml 触发分支 / runner / Node / pnpm 版本

### V1.3 提议

### 实现落点(V1.4-1)

- `Object.keys(res.body.data)` 3 处补 `as object` 局部断言,消除 supertest `Response.body: any` 在 `Object.keys` 参数位的 4 个 warning ✅
- `package.json#scripts.lint` 改为 `eslint --max-warnings 0 ...`,本地 / CI 共用同一入口 ✅
- `pnpm lint` 输出 `0 errors / 0 warnings`;`pnpm test` / `pnpm test:contract` / `pnpm test:e2e` / `pnpm build` 全绿 ✅
