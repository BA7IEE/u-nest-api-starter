# V1.4-2 Prisma 7 升级评估

> 状态:**仅评估,不升级依赖,不改运行时代码**。
> 立项前提:`v0.1.4` 已发布,V1.4-1 lint 严格模式已合并(commit `9b02670`)。
> 范围承诺:本任务仅交付本评估文档;升级动作(若决定执行)按 §6 拆分为独立任务,**不在 V1.4-2 内完成**。
> 升级判定原则:沿用 [`ARCHITECTURE.md`](../ARCHITECTURE.md) §9——"业务诉求驱动,不为升级而升级"。

## 0. 一句话结论

**当前不建议升级到 Prisma 7。** 当前 Prisma 6.19.3 + `prisma-client-js` 在本仓库工作完全正常,Prisma 7 强制带来的 generator 迁移(`prisma-client-js` → `prisma-client`)会**联动改写 Dockerfile 多阶段构建中已经手工裁剪过的 prod 子集复制逻辑**(见 [`Dockerfile`](../Dockerfile) §80-§150),投入产出比低。建议在第一个真正需要 Prisma 7 新特性(如新驱动适配、ESM 输出、新查询能力)的业务诉求出现前**保持 6.x**,仅做最小化的 deprecation 收敛(§6.1)。

## 2. Prisma 7 breaking changes 与本仓库受影响面

依据 Prisma 官方升级指南整理。每条标记 **影响等级**(高 / 中 / 低 / 无)与 **是否阻塞升级**。

### 2.1 `package.json#prisma` 配置块移除 → 必须迁到 `prisma.config.ts`

- **影响等级:中**
- **阻塞升级:是**(Prisma 7 不再读取 `package.json#prisma`)
- **本仓库现状**:`package.json#prisma.seed = "tsx prisma/seed.ts"`(见 [package.json:31-33](../package.json:31))
- **改动范围**:删除 `package.json` 中 `prisma` 块,新增 `prisma.config.ts`,内容大致为:

  ```ts
  import type { PrismaConfig } from 'prisma';
  export default {
    migrations: { seed: 'tsx prisma/seed.ts' },
  } satisfies PrismaConfig;
  ```

- **副作用**:`prisma.config.ts` 必须在仓库根目录,会被 TypeScript / ESLint 扫到,需评估是否纳入 `tsconfig.json` 与 `eslint.config.mjs` 的 `project` 列表。
- **可在 Prisma 6.x 提前迁**:Prisma 6.x 已经支持 `prisma.config.ts`(`package.json#prisma` 在 6.x 是 deprecation,在 7.x 是 removal),迁移可作为 Prisma 7 升级的"前置 PR",降低主升级 PR 的风险面。

### 2.2 Generator `prisma-client-js` deprecated → 建议迁到 `prisma-client`

- **影响等级:高**
- **阻塞升级:否**(Prisma 7 仍兼容 `prisma-client-js`,但官方明确"deprecated",未来某个主版本会移除)
- **`prisma-client` 的不同**:
  - `output` 字段**必填**(没有默认 `node_modules` 出口);典型值如 `./generated/prisma/client`
  - 生成产物是**普通 TS / JS 源码**(不在 `node_modules`),随业务一起被打包
  - 支持 ESM / CommonJS via `moduleFormat`
  - 运行时**不再自动加载 `.env`**;需要调用方自行 `dotenv.config()` 或保证 `process.env` 已注入
- **本仓库受影响面**:
  - `prisma/schema.prisma` 的 `generator client` 块要改 `provider` 与新增 `output`
  - 4 个 `@prisma/client` import 必须全部改成 `output` 路径(或保留一个 re-export 桥接文件,但桥接文件会和 `output` 路径绑定耦合,长期不优雅)
  - **Dockerfile §80-§150 的 prod 子集裁剪逻辑要重写**:当前逻辑假设 generator 产物在 `node_modules/.pnpm/@prisma+client@.../node_modules/{@prisma/client, .prisma}`,迁到 `prisma-client` 后产物在 `./generated/...`(源码目录),不再需要"备份 → 重装 → 还原"的体操,但 `dist/` 中的导入路径会随 `nest build` 解析到不同目录,需要重新核对镜像 smoke test
  - `users.select.ts` 中 `Prisma.UserSelect` / `Prisma.UserGetPayload` 这类 namespace 类型在 `prisma-client` 下仍可用(类型 namespace 不变),但导入路径要改
- **运行时 `.env` 加载差异**:本项目运行时 `DATABASE_URL` 由 NestJS `ConfigModule` 注入(在 PrismaClient 构造前),不依赖 `prisma-client-js` 自动 dotenv,**不构成回归点**。`prisma/seed.ts` 通过 `prisma db seed` 调起,Prisma CLI 自身仍会加载 `.env`,也不构成回归点。

### 2.3 `PrismaClientKnownRequestError` / P2002 / `err.meta?.target` 形状

- **影响等级:无**
- **阻塞升级:否**
- **官方升级指南未提及该 error 类的 breaking change**。本仓库 [users.service.ts:100](../src/modules/users/users.service.ts:100) 的处理方式(`err instanceof Prisma.PrismaClientKnownRequestError && err.code === 'P2002'` + `err.meta?.target as string[]`)在 Prisma 7 仍然有效。
- **验收点**:升级后 `pnpm test:e2e` 中 `users-admin-crud.e2e-spec.ts` 的 `username already exists` / `email already exists` 用例必须继续 409,响应体 `{ code: 10002 / 10003 }` 不变。

### 2.4 Prisma CLI 命令行行为变化

- **影响等级:低**
- **阻塞升级:否**
- **变化**:
  - `prisma migrate dev` 不再自动 `prisma generate`(本仓库 CI 与 Dockerfile 都已**显式** `pnpm prisma:generate`,不依赖隐式行为)
  - `prisma migrate dev` 不再自动跑 seed(本仓库 seed 通过独立 `pnpm prisma:seed` 运行,不依赖 migrate dev 联动)
  - `--skip-generate` / `--skip-seed` flags 被移除(本仓库未使用)
  - `prisma db execute` 移除 `--schema` / `--url` flags(本仓库未使用 `db execute`)
- **结论**:本仓库 CLI 用法已经显式化,不受本组改动影响。

### 2.5 Node.js 最低版本提升至 `20.19.0+`

- **影响等级:无**
- **阻塞升级:否**
- 本仓库 `package.json#engines.node = ">=22"`,Dockerfile `NODE_VERSION=22`,CI `setup-node` 用 `node-version: 22`。**全部高于 20.19**,无回归。

### 2.6 Query engine binaryTargets / alpine / OpenSSL

- **影响等级:低**
- **阻塞升级:否**
- 官方升级指南未声明 alpine / `linux-musl-openssl-3.0.x` 二进制目标变化。Dockerfile 已显式 `apk add libc6-compat openssl`,与 Prisma 6 一致。
- **验收点**:升级后必须重新 `docker build` 并跑容器 smoke test(`GET /api/health`、`/api/health/ready`、`/api/docs`),确认 query engine `.so` 在 `node:22-alpine` 下能加载。

### 2.7 Seed 配置语法

- **影响等级:中**(与 §2.1 是同一件事)
- 老:`package.json#prisma.seed = "tsx prisma/seed.ts"`
- 新:`prisma.config.ts` 中 `migrations.seed = "tsx prisma/seed.ts"`

### 2.8 pnpm / packageManager 兼容性

- **影响等级:无**
- 本仓库 `packageManager: "pnpm@10.14.0"`,Prisma 7 未声明对 pnpm 的最低版本要求,`pnpm install --frozen-lockfile` 行为不受 Prisma 主版本影响。

### 2.9 NestJS / `@nestjs/common` 接入面

- **影响等级:无**
- `PrismaService extends PrismaClient implements OnModuleInit, OnModuleDestroy`(见 [prisma.service.ts](../src/database/prisma.service.ts))在 Prisma 6 / 7 都成立。NestJS `OnModuleInit` / `OnModuleDestroy` 与 Prisma 主版本解耦。

## 4. 推荐升级步骤(若未来决定执行)

> 以下顺序基于"小步、可回滚、每步独立 review"原则。每个步骤是**独立 PR**。

### Step 1(仍在 Prisma 6.x):`prisma.config.ts` 迁移 [done — V1.4-3]

1. 新建 `prisma.config.ts`,把 `package.json#prisma.seed` 内容迁过去 ✅
2. 删除 `package.json` 中 `prisma` 块 ✅
3. 验证 `pnpm prisma:seed` / `pnpm prisma:migrate` / `pnpm prisma:deploy` 全部成立 ✅(generate / deploy / seed 三命令均输出 `Loaded Prisma config from prisma.config.ts.`,seed 幂等性二次验证亦通过)
4. ⚠️ Prisma CLI 检测到 `prisma.config.ts` 后会**关闭**自动 `.env` 加载(输出 `Prisma config detected, skipping environment variable loading.`),为保持本地开发流程不变,在 config 顶部 `import 'dotenv/config'`(`dotenv` 已是 devDependency,无新增依赖,lockfile 不漂移);CI / 容器构建因为不依赖 `.env` 文件,silent-skip 是安全的
5. **未**把 `prisma.config.ts` 加进 `tsconfig.json#include`:Prisma CLI 自带 TS 加载器(`jiti` / `c12` 内置),不依赖 `pnpm typecheck`;加进 include 会让该文件被 `nest build` 编译进 `dist/`,污染产物,得不偿失
6. **未**改 `eslint.config.mjs#project` 列表:`prisma.config.ts` 在仓库根目录,本就不在现有 lint glob(`src/**` / `test/**` / `prisma/**`)内,无需调整
7. 跑 lint / typecheck / test / test:contract / test:e2e / build 全绿 ✅
8. **不动**版本号、不动 Dockerfile、不动 CI、不动 Prisma schema(因为 Prisma 仍是 6.x)✅

### Step 2:bump `prisma` + `@prisma/client` 到 7.x,**保留** `prisma-client-js` generator

1. `pnpm add -D prisma@7` + `pnpm add @prisma/client@7`
2. `pnpm prisma:generate` 重新生成
3. 跑全套验收:lint / typecheck / test / test:contract / test:e2e / build
4. **本地** `docker build` 一次,跑容器 smoke test(`/api/health` / `/api/health/ready` / `/api/docs`)
5. 按 Dockerfile §118-§129 维护铁律,重新核对 `RUN rm -rf` 清单(主要看 `@prisma/engines*` / `@prisma/config` / `effect` 是否有结构变化)
6. CI 镜像 build job 必须绿
7. 版本号 bump:`package.json#version` + Swagger `setVersion(...)`,`CHANGELOG.md` 记录"Prisma 7 升级,保留 deprecated generator"

### Step 3(可选,延后):generator 迁到 `prisma-client`

> 这一步的成本远高于 Step 1+2 之和。只有在以下任一条件成立时才做:
> - 业务确实需要 `prisma-client` 的 ESM / 自定义 output / TS 源码可调试 等新特性
> - Prisma 8.x 即将彻底移除 `prisma-client-js`,被动迁移
>
> 在那之前**不主动做**。

1. 在 `schema.prisma` 改 `generator client { provider = "prisma-client", output = "./generated/prisma/client" }`
2. 改 4 处 `from '@prisma/client'` 到 `from '<output>/client'`
3. 重写 Dockerfile §80-§150 的 prod 子集裁剪逻辑(generator 产物变成源码目录,不再需要"备份 → 重装 → 还原"体操)
4. 调整 `.gitignore` / `eslint.config.mjs` ignore patterns,避免 generated TS 源码被 lint
5. 镜像 smoke test 全套
6. CHANGELOG 显式说明 import 路径变化(虽对外部 API 无影响,但对二开者有影响)

### Step 4(可选):清理 + 文档

- 更新 `docs/development.md` 中"`@prisma/client`"相关表述
- 更新 `docs/deployment.md` 中"runner 阶段 Prisma CLI"相关表述
- 更新 `ARCHITECTURE.md` 若涉及 generator 选型说明

## 5. 回滚方案

每个步骤都是独立 PR,git revert 即回滚:

| 失败位置 | 回滚动作 |
|---|---|
| Step 1 失败(`prisma.config.ts` migration) | `git revert` Step 1 commit;`package.json#prisma` 段恢复 |
| Step 2 失败(7.x 升级) | `git revert` Step 2 commit;`pnpm install --frozen-lockfile` 回到 6.19.3 lockfile;若 lockfile 也已 commit,reset 到 main 上一版 |
| Step 3 失败(generator 迁移) | `git revert` Step 3 commit;**注意 generated 文件**:若 generated 目录已入 git,revert 自动恢复;若用 `.gitignore` 排除则需手动 `pnpm prisma:generate` 一次 |
| 容器运行时失败(Dockerfile rm 清单错) | 临时回退 image tag 到上一版;按 Dockerfile §118-§129 铁律重新核对清单后再发 |
| 数据库不可逆变化 | **本评估范围内 Prisma 7 升级不要求 schema 变化**;若误执行 `prisma migrate dev` 生成新 migration,删除 migration 目录 + DB 端 `_prisma_migrations` 对应行(开发库)/ 联系 DBA(生产) |

回滚关键前提:**升级 PR 不绑定业务 schema 变更**(本评估明令 Step 2 / Step 3 不动 `prisma/schema.prisma`)。

## 6. 是否建议现在升级

**不建议现在升级到 Prisma 7。**

判定依据:

1. **业务驱动缺失**:本仓库目前不需要 `prisma-client` 的 ESM 输出、自定义 output、新驱动适配。Prisma 6.19.3 在 P2002 / 软删除 / 事务 / migrate deploy 全部场景工作正常,162 个 e2e 用例全绿。
2. **联动改动面大**:Step 3(`prisma-client-js` → `prisma-client`)涉及 `prisma/schema.prisma` + 4 处源码 import + Dockerfile §80-§150 重写 + smoke test 全套。当前 V1.4 阶段定位是"模板期收敛",大改不匹配阶段定位。
3. **风险收益不对称**:维持 6.x 的成本仅是启动期一行 `Update available` 提示;升级的成本是 R1-R8 风险矩阵 + 一次完整镜像 / CI 验证周期。
4. **Prisma 7 兼容窗口足够**:`prisma-client-js` 在 7.x 仍兼容,只是 deprecated;真正的 forced migration 至少要等 8.x。

### 6.1 唯一例外:可考虑现在做的最小化收敛 [done — V1.4-3]

**`package.json#prisma` → `prisma.config.ts` 迁移**(即 §4 Step 1)可以现在做,理由:

- 在 Prisma 6.x 内即可完成,与版本号无关
- 消除 6.x 启动期的 deprecation warning 噪声
- 为未来 Prisma 7 升级提前清出"格式包袱",降低主升级 PR 的 diff
- 改动只动 `package.json` 与新增 `prisma.config.ts`,blast radius 极小

**已在 V1.4-3 执行完成**(详见 §4 Step 1 实施说明)。后续若决定继续 Prisma 7 升级,从 §4 Step 2 开始。
